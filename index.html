01 · Multi-Agent 编排引擎

为什么选 LangGraph 而不是 LangChain

LangChain 的 AgentExecutor 封装了完整的 ReAct 循环——调 LLM→有 tool_call→执行→结果塞回→重复→Final Answer。问题是这个循环是黑盒。我要在"工具执行完"和"返回结果"之间加 Reflection 评分和条件重试——AgentExecutor 的源代码里没有这个钩子。

LangGraph 把每一步拆成独立节点——StateGraph 上用 7 个节点（classify + 5 Agent + reflect），条件边做路由，循环边做重试。每一步 LLM 调用的消息拼接都在我手里。

选型逻辑：不是 LangGraph 比 LangChain 好——是我的场景需要条件路由和循环边，这两个是 LangGraph 的核心能力，LangChain 做不到。场景决定技术选型。

7 节点图结构

用户输入 → classify（意图+ RAG+ Mem0）
            │ 条件边(routeTo)
            ▼
  concept / exam / code / plan / news
            │
            ▼
          reflect（四维评分）
            │ 条件边(needsRetry?)
            ├─ true → 退回 Agent
            └─ false → END

State 设计

const AgentState = Annotation.Root({
  messages: Annotation<BaseMessage[]>({
    reducer: (c, u) => c.concat(u),  // 追加，不覆盖
    default: () => [],
  }),
  routeTo: Annotation<string>(),        // lastValue 覆盖
  reflectionScore: Annotation<number>(),
  needsRetry: Annotation<boolean>(),
  ragContext: Annotation<string>(),
  memoryContext: Annotation<string>(),
});
踩坑：一开始 messages 和 routeTo 都用了 concat reducer——routeTo 变成数组 ["concept","concept"]，条件边无法匹配字符串，所有路由全乱了。区分"累积型状态"和"当前型状态"是 LangGraph StateGraph 的核心设计。

Reflection 自纠错

四维评分: 准确率(0.35) + 清晰度(0.30) + 可运行性(0.15) + 满意度(0.20)
评估: LLM 输出 JSON {score, issues, shouldRetry, retryStrategy}
修正: score < 0.75 → 诊断问题 → 修正重答 → 最多 3 轮
实现: LangGraph 循环边——条件边函数控制 needsRetry && retries < 3
Q: LLM 自己评自己，评错了怎么办？
这是 Reflection 的已知局限。两个补偿：准确率权重最高(0.35)逼 LLM 优先检查事实；最多 3 轮防止无限循环。生产环境加外部验证——搜索引擎核实每个声明来源。当前规模 RAG+Reflection 两层把关够用。

02 · RAG 检索增强全链路

离线入库

知识条目 → 递归切分(优先\n\n→\n→。边界) → 父子文档(300tk/2000tk)
  → DeepSeek Embedding(1536维) → LanceDB + sql.js 元数据

为什么递归切分不是固定长度：固定 500 字切一刀可能在"GRPO 是 DeepSeek 提出"和"的算法"中间切断——后半段碎片在向量库里毫无意义。递归切分优先找自然边界，零成本。

在线检索：两阶段

第一阶段: LanceDB 向量 + sql.js 关键词 → RRF 融合(k=60) → top 15
第二阶段: BGE Reranker V2-M3 Cross-Encoder 精排 → top 5
精排全程在子文档层面操作 → parent_id 展开父文档 → 注入 LLM Prompt

Cross-Encoder 为什么比向量检索准：向量检索是"分别编码、再算距离"——问题向量和文档向量之间没有交互。Cross-Encoder 是"拼接编码、一起算分"——把"问题+文档"拼成一句话过 24 层 BERT，跨句注意力能判断"问题里的 GRPO 和文档里的 GRPO 是不是同一个东西"。

Q: Cross-Encoder 这么准，为什么不能替代向量检索搜全库？
不能建索引。全库 500 条文档，每条都和问题拼接过一次 BERT——500×50ms=25 秒。向量检索 O(log n) 先砍到 15 条，再用 Cross-Encoder 慢慢挑 5 条。两阶段各司其职。
Q: RRF 的 k=60 哪来的？
不是自己调的——LlamaIndex 和 Weaviate 都用 60。k=0 时第一名 1/1=1.0，第五名 1/5=0.2——差距太大。k=60 做平滑：1/61≈0.0164，1/65≈0.0154——排名靠前的文档分数接近，不因为某一次排名就淘汰好结果。

03 · Mem0 记忆系统设计

为什么不用 LangGraph 原生 Store，选了 Mem0

LangGraph 原生 Store 是 Key-Value 存储——需要手动 Put/Get。Mem0 能让 LLM 自动从对话中提取记忆单元：用户说"我学过 RAG，但不熟悉 HNSW"→自动生成两条记忆 {"已掌握 RAG", "HNSW 薄弱"}。不需要预定义 Schema。

冲突检测是 Mem0 的核心能力——用户后来说"我其实不懂 RAG"，Mem0 会自动覆盖之前"已掌握 RAG"的记忆。

记忆工作流

classify 节点:
  1. Mem0 语义检索——召回与当前问题相关的记忆
  2. 记忆内容写入 State.memoryContext
  3. 后续所有 Agent 节点从 State 读取——不需要各自查 Mem0

回答后更新:
  1. 根据 Reflection 评分自动更新知识掌握度
  2. 评分 < 0.4 → 加入薄弱点
  3. 评分 > 0.8 → 加入已掌握

04 · MCP 工具协议与容错

MCP 架构

MCP Host (你的应用)
  ├── MCP Client → JSON-RPC → Server 1: 代码执行
  ├── MCP Client → JSON-RPC → Server 2: arXiv 检索
  ├── MCP Client → JSON-RPC → Server 3: 知识库搜索
  └── MCP Client → JSON-RPC → Server 4: 学习进度
Q: 为什么用 MCP 不是 Function Calling？
Function Calling 工具定义写在代码里→加工具改代码重启服务。MCP 工具是独立进程→热插拔→跨框架复用（LangChain/AutoGen/自研都能调）。最关键的——MCP Server 可跑在 Docker 容器里做安全隔离——代码执行这个 Server 必须隔离。

三层错误处理

第一层: 工具函数内部——参数校验 + try-catch（即时）
第二层: ToolNode——raise_on_error=false 容错执行（秒级）
第三层: 条件边恢复路由（流程级）
        重试<3次 → 退回 agent 重试
        重试≥3次 → 降级回答（基于已有结果给部分答案）
        严重错误 → 终止并告知用户
最大安全隐患：代码执行当前是裸 exec()。设计用 Docker 隔离——非 root 用户 + --memory=256m --network=none。即使 exec 被恶意利用也只拿到受限权限。

06 · Runtime 三层分离

State   ← 放请求级动态数据（messages, routeTo, reflectionScore）
          reducer 控制合并策略

Store   ← 放跨会话持久化数据（用户画像、学习记录）
          Mem0 实现，按需读写

Context ← 放运行时静态配置（user_id, max_tool_calls）
          graph.invoke() 时通过 configurable 注入
          只读——不进序列化、不注入 LLM Prompt
为什么要分：API Key 放进 State→被序列化到 LangSmith Trace 日志→安全风险。user_id 放进 State→被注入 LLM System Prompt→浪费 Token。Context 只读注入→不进任何序列化链路。

解析引擎核心实现

解析器是整个项目的入口——输入 trajectory.json 的完整 JSON 对象，输出结构化的 ParsedStep[] 数组。要同时处理两种 content 格式、<thinking> 标签提取、工具名称归一化。

// 核心解析逻辑（简化）
function parseTrajectory(json) {
  const steps = []
  
  for (const msg of json.trajectory) {
    if (msg.role === "assistant") {
      if (typeof msg.content === "string") {
        // 场景1: 纯文本 → 拆分 thinking + text
        const thoughts = extractThoughtBlocks(msg.content)
        thoughts.forEach(t => steps.push({ type: "thought", content: t }))
        // 剩余非 thinking 文本
        const rest = msg.content.replace(/<thinking>[\s\S]*?<\/thinking>/g, "").trim()
        if (rest) steps.push({ type: "text", content: rest })
      } 
      else if (Array.isArray(msg.content)) {
        // 场景2: 多 block → text / tool_use
        for (const block of msg.content) {
          if (block.type === "tool_use")
            steps.push({ type: "tool_call", toolName: block.name, ... })
          else if (block.type === "text")
            steps.push({ type: "text", content: block.text })
        }
      }
    }
  }
  return steps
}
踩坑：content 可能是 string 或 Array<object>——同一个 Assistant 消息的 content 就有两种形态。一开始写死了 typeof === "string" 的路径，Array 格式的消息全部跳过。工具调用在时间线上不显示，排查了半小时才发现是类型分支遗漏。

知识图谱：dagre + ReactFlow 布局

为什么不是 echarts/antv：两者的图布局是声明式 option 配置，不支持自定义 React 组件作为节点。我需要工具调用节点上显示参数摘要、文件节点显示路径、点击跳转步骤——这些都是 React 组件才能实现的交互。ReactFlow 每个节点是完整 React 组件。

// graph-builder 核心：steps → dagre 布局节点
function buildGraph(steps, currentStep) {
  const g = new dagre.graphlib.Graph()
  g.setGraph({ rankdir: "TB", ranksep: 80, nodesep: 60 })
  
  steps.forEach((step, i) => {
    const nodeId = `step-${i}`
    g.setNode(nodeId, { width: 200, height: 80 })
    
    // tool_call → 读取 filePath → 创建文件节点 + 边
    if (step.type === "tool_call") {
      const filePath = extractFilePath(step)
      if (filePath) {
        g.setNode(fileId, { width: 180, height: 60 })
        g.setEdge(nodeId, fileId) // tool→file 边
      }
    }
  })
  
  dagre.layout(g) // 自动计算 x/y 坐标
  return { nodes, edges }
}
Sugiyama 算法原理：dagre 的四步流程——(1) 去环：反转边消除有向环 (2) 分层：每个节点分配层号 rank (3) 排序：层内最小化交叉 (4) 定位：计算精确 x/y。graphlib 提供 setNode/setEdge API，dagre.layout() 原地写入坐标。
Q: 节点数超过 100 个，dagre 布局会卡吗？
dagre 时间复杂度 O(N²·L)（N=节点数，L=层数）。100 节点约 5ms 完成。瓶颈不在 dagre 而在 ReactFlow 的 DOM 渲染——解决方案是 ReactFlow 自带的节点虚拟化（nodesDraggable=false + onlyRenderVisibleElements=true），视口外节点不渲染 DOM，1000 节点也不卡。

播放状态机

// Zustand Store 中的播放逻辑
useEffect(() => {
  if (isPlaying) {
    const ms = 2000 / playbackSpeed  // 1x=2s/步, 4x=0.5s/步
    timerRef.current = setInterval(nextStep, ms)
    return () => { if (timerRef.current) clearInterval(timerRef.current) }
  }
}, [isPlaying, playbackSpeed])

// nextStep: 步进 + 自动暂停
nextStep: () => {
  if (currentStep < totalSteps - 1) set({ currentStep: currentStep + 1 })
  else set({ isPlaying: false })  // 到末尾自动暂停
}
踩坑：React 19 中 useRef 签名变了——useRef<ReturnType<typeof setInterval>>() 报 Expected 1 arguments but got 0。因为 React 19 的 useRef 重载强制要求 initialValue。改为 useRef<NodeJS.Timeout | null>(null)，同时 useEffect cleanup 里判空再 clearInterval。

工程落地中的实际问题

问题1: ReactFlow 自定义节点投产后白色边框闪烁
ReactFlow 内部用 React.memo 做节点 diff。自定义 AgentNode 传入的数据对象引用每次都变——Memo 失效，节点重新渲染。解决方案：useMemo 稳定 nodes 数组引用 + AgentNode 的 data 字段只传必要字段（用对象展开会导致引用变化）。
问题2: trajectory 中 tool_use 的 id 和 tool_result 的 tool_use_id 不匹配
解析器对两者不做硬校验（不匹配就跳过），但 Dashboard 统计中会标记"孤立调用"数量。真正的自洽性校验应该在数据生产环节做——标注平台入库前检查。
问题3: Monaco Editor 在生产环境下加载 2-4 秒白屏
Monaco 的 Worker 是异步 CDN 加载。SSG 首屏不渲染 Diff 视图（tab 切换时才挂载），首屏性能不受影响。如果要进一步优化：next.config 中配置 transpilePackages + 预加载 Monaco Worker URL。
问题4: 样本数据中 final_diff 的补丁格式不完全标准
GitHub PR 的 unified diff 有时包含 "No newline at end of file"、空 hunk、图片 binary diff 等非常规内容。当前的简易 parser 遇到这些会退化——跳过异常 hunk 直接展示原始内容。生产级方案应使用 diff 库（npm: diff）做规范 parse。
问题5: 文件节点如果提取不到 filePath 怎么办
extractFilePath 从 toolInput 中依次尝试 filePath → path → file → pattern（grep 用 pattern 而非 filePath）。都为空就跳过——不创建文件节点。这会导致图中某些 tool_call 节点没有文件关联边，不影响核心功能。

面试追问预判

Q: 你这个项目跟普通的 SWE-bench viewer 有什么本质区别？
SWE-bench viewer 看的是"任务结果"——这个 issue 修没修好。CodeTrail 看的是"过程"——Agent 的每步思考、每个工具调用、文件修改的因果链。核心价值是可解释性：Agent 为什么改了 this file 而不是 that file？为什么在这里调了 3 次 read_file？所有推理路径可视化。
Q: 知识图谱的布局算法你能讲讲原理吗？
dagre 用的是 Sugiyama 分层布局——四步：(1) 去环——DFS 反向边消除有向环 (2) 分配层号——最长路径算法确保边只向下 (3) 层内排序——barycenter 重心法最小化交叉 (4) 坐标计算——brandes-köpf 快速水平坐标。dagre 选择了 O(N²·L) 而非 O(N³) 的实现，100 节点内 5ms 布局完成。
Q: 为什么要用 Next.js 做这个项目？纯 React 不就行了吗？
当前确实是 100% Client Component，Next.js 的服务端能力没用到。选 Next.js 是为后续扩展预留空间——计划加入 Server Action 做 trajectory 质量自动评分（调 AI 接口打分）、RSC 渲染分享页。如果只是原型，Vite 更轻——但这个项目定位是生产级工具。
Q: Zustand 比 Redux 好在哪？你不怕全局状态混乱吗？
Zustand 的 create() 返回 hook，组件里 usePlaybackStore(s => s.currentStep) 精确订阅单一字段——不像 Redux 的 useSelector 需要手动 shallowEqual。全局状态在这里是合理的——只有一个 trajectory 数据源，不是多页面多表单的复杂场景。如果以后加多个 trajectory 并行对比，再考虑拆 store。
Q: 这个项目的数据从哪里来？能对接真实 Agent 吗？
当前数据入口是 trajectory.json 文件上传。数据格式遵循 Anthropic Messages API + git_context 扩展字段——这是 SWE-bench 数据标注团队的输出规范。对接真实 Agent（如 Claude Code / CLINE）需要加个 adapter 层做格式转换——把 Agent 的原生日志转成 trajectory.json。技术难度不高，主要工作量在字段映射。
Q: 性能方面你有什么考虑？
四个关键点：(1) Monaco Editor 懒加载——只在切换到 Diff tab 时才 dynamic import (2) ReactFlow 节点虚拟化——视口外节点不渲染 DOM (3) 知识图谱布局用 useMemo 缓存——只有 steps 变化才重新计算 dagre (4) 播放引擎用 setInterval 而非 requestAnimationFrame——不需要逐帧动画，步骤粒度足够。实测 100 步轨迹 timeline 渲染 < 50ms，图谱布局 < 5ms。
