一、Agent 基础概念

Q1什么是 Agent，和普通 LLM 调用有什么区别？
普通 LLM 调用是一次性的——你问它答，它只能给你训练数据里有的信息。

Agent 是 LLM + 工具 + 自主决策循环。它会在每轮自己判断"信息够不够？不够我该调什么工具？"——然后调工具、拿结果、再判断。这个循环直到它认为任务完成了才结束。

打个比方：普通 LLM 像问路——"故宫怎么走"，它凭记忆告诉你。Agent 像给你一个助手——"帮我去故宫"，它会自己查地图、规划路线、打车、路上遇到封路自己绕行，直到把你送到。
Q2什么是 ReAct？
ReAct = Reasoning + Acting，2022 年 Google 提出的一种 Agent 推理范式。

核心流程：Thought（思考）→ Action（行动）→ Observation（观察）→ Thought → Action → ... → Answer

跟普通 LLM 的区别：普通 LLM 直接凭着训练数据回答（可能过时或幻觉）。ReAct 强制模型"先想再动再看结果"，每一步推理有据可查，减少胡说八道。你标注的 trajectory.json 里 <thinking> 就是 Thought，tool_use 就是 Action，tool_result 就是 Observation。
Q3Function Calling 和 Tool Use 有什么区别？
本质是一个东西——让 LLM 返回结构化 JSON，告诉程序"我要调这个函数/工具，参数如下"。不同厂商有不同叫法：Function Calling 是 OpenAI 的叫法，Tool Use 是 Anthropic 的叫法。

需要特别注意的是：不是 LLM 真的在执行工具。LLM 只输出调用指令，你的程序去执行，然后把结果塞回 LLM。
Q4什么是 CoT（思维链）？
CoT = Chain of Thought，让 LLM 把推理过程一步步写出来，而不是直接给答案。

不用 CoT："11 是质数吗？" → "是"
用 CoT："质数定义是只能被 1 和自身整除。检查 11 能否被 2~10 整除：2 不行、3 不行...所以 11 是质数。"

准确率显著提升，每一步可审计，中间步骤错就能发现。你标注的 trajectory.json 里 <thinking>...</thinking> 就是 CoT。
Q5CoT、ReAct、Plan-and-Execute 三者的区别是什么？
三者都是 Agent 的推理范式，复杂度递进：

CoT——无工具调用，把思路写出来，纯推理
ReAct——想一步、做一步、看结果、再继续想，适合需要查资料/调工具的简单任务
Plan-and-Execute——先列计划再逐步执行，适合多步骤复杂任务

实际工作场景中通常一个项目里这三种范式我们都会用到：用 Plan-and-Execute 拆任务，单个子任务内部用 ReAct 做探索，每一步推理都用 CoT 降低出错率。
Q6什么是 Reflection 设计模式？
Reflection = LLM 自己评自己，打分不合格就重做。

流程：模型输出 → 交给 LLM 评分 → score < 阈值 → 拿反馈去修正 → 重新输出 → score >= 阈值 → 通过

在 LangGraph 里的实现：reflect 节点做四维评分（准确率 + 清晰度 + 可运行性 + 满意度），score < 0.75 就通过循环边退回重做，最多 3 轮。这就是 Reflection 的工业级落地。

二、多 Agent 编排

Q7什么时候用多 Agent，什么时候用单 Agent？
判断标准：看任务能不能拆成独立子任务、分工是否带来明显质量提升。

多 Agent 的价值 = 分工带来的质量提升 − 编排带来的额外复杂度。如果分工不明显、不需要并行、不需要角色隔离（不同 system prompt），单 Agent 就够了。

✅ 该用多 Agent：上线新功能 → A 读需求、B 写代码、C 审查、D 测试，不同 Agent 有不同 system prompt
❌ 不该用：修一个参数缺失的 Bug → 一个 Agent 读-改-测三步闭环
Q8Orchestrator 模式和 Peer-to-Peer 模式的区别？
Orchestrator（编排者）：中心化，一个老板指挥多个员工。classify 是唯一路由决策点，Agent 之间不通信。可控，debug 一条线。

Peer-to-Peer（对等）：去中心化，每个 Agent 自己决定跟谁说话。灵活，动态协作，不怕单点故障。但容易聊飞了，收敛困难。

我们的 agent-notes 项目是 Orchestrator——classify 按意图路由到 concept/exam/code 等子 Agent，Agent 之间不直接通信。
Q9多 Agent 之间怎么通信？
三种方式，按复杂度递增：

1. 共享 State——所有 Agent 读/写同一个 State 对象（LangGraph 的做法，最简单）。Agent A 输出写 State，Agent B 从 State 读。

2. 消息传递 Send/Receive——有明确收发关系，不污染全局 State。适合一对一协作。

3. 消息队列/总线——所有 Agent 往 topic 发消息，关注该 topic 的 Agent 自己收。适合广播场景。
Q10多 Agent 怎么通过 State 共享数据？
就是公共黑板，不是互相喊话。以 LangGraph 为例：

classify 执行完 → 写入 State: { messages: [...], routeTo: "code", ragContext: "..." } → LangGraph 路由到 code Agent → code 从 State.messages 读前面所有对话历史 → 生成回答，追加到 State.messages → reflect 执行，读所有消息 → 打分。

关键机制是 reducer（合并策略）：messages 用 concat 追加（历史越滚越长），routeTo 用 lastValue 覆盖（新值覆盖旧值）。
Q11并发写 State 怎么处理冲突？
LangGraph 天然就没有并发写冲突——因为节点是顺序执行的，同一时刻只有一个节点在跑。

但在 Plan-and-Execute 中用 Send() API 并行执行无依赖子任务时：并行执行 → 各自产生输出 → Fan-in 等待全部完成 → reducer 合并。

冲突处理靠的就是 reducer：messages 用 concat 全部保留，routeTo/score 用 lastValue 最后完成的生效，还可以自定义 reducer（比如取最高分：Math.max）。
Q12什么情况下多 Agent 可以并行，什么情况下必须串行？
一条规则：子任务之间有依赖就必须串行，没依赖就能并行。

判断标准：
• 数据依赖：Agent B 是不是要用 Agent A 的输出当输入？
• 状态依赖：Agent B 是不是等 Agent A 改完代码才能动手？

任一答"是" → 串行。都"否" → 并行。例如：写 README、写测试、写 Dockerfile 三者互不依赖，能并行。读源码 → 改代码 → 跑测试有严格依赖，必须串行。

三、框架与协议选型

Q13LangGraph、AutoGen、CrewAI 分别适合什么场景？
LangGraph——Agent 编排引擎，精细化控制每一步走向。适合复杂工作流、条件路由、循环。
AutoGen——多 Agent 对话框架，适合代码审查（2 Agent 互相挑错）、头脑风暴。
CrewAI——角色扮演式 Agent 团队，适合快速搭 CEO/程序员/测试员团队。上手难度最低。

实际项目通常是混用的：用 CrewAI 定义角色分工，遇到需要循环自纠错的场景，在单个 Agent 内部用 LangGraph 的循环边实现。互不冲突。
Q14MCP 是什么，解决了什么问题？
MCP = Model Context Protocol，Anthropic 提出的标准化工具调用协议。

以前不同框架调同一套工具要写不同的代码——LangChain 一套、AutoGen 一套、自研框架又一套。MCP 把框架和工具解耦：工具（MCP Server）写一次，任何框架的 MCP Client 都能调，支持热插拔，Server 可跑在 Docker 里做安全隔离。

一个类比：MCP 是工具的 USB 接口——定义统一协议，插上就能用。
Q15A2A 协议是什么？同一个系统内需要吗？
A2A = Agent-to-Agent Protocol，Google 提出，不同系统之间 Agent 互相委派任务的标准协议。核心三要素：Agent Card（能力名片）、Task（任务对象）、协商机制。

同一个系统内完全不需要 A2A。LangGraph 里 Agent A 往 State 写、Agent B 读——都在一个进程里。MCP 管手（调工具），A2A 管嘴（跨系统）。只有当你的 Agent 要去调别人写的 Agent（跨进程、跨团队、跨公司）时，才需要 A2A 做标准化的握手和委派。

四、记忆系统

Q16Agent 的四种记忆类型分别是什么？
感官记忆——当前对话窗口，单次上下文。"刚才用户说了什么"
短期记忆——当前会话，一轮对话。"几分钟前讨论过什么"
长期记忆——外部存储（向量库/数据库），跨会话永久。"用户学过 RAG，HNSW 薄弱"
工作记忆——临时草稿区，单次推理。"当前任务拆出的子步骤进度"

对应项目里的实现：感官记忆 = State.messages；短期记忆 = LangGraph Checkpointer；长期记忆 = Mem0；工作记忆 = State.subTasks + reflectionRetries。
Q17Context Window 满了怎么办？
三种策略：

1. 滑动窗口——保留最近 N 条，超出的丢掉。最常用。
2. 摘要压缩——用 LLM 把旧消息压缩成一段摘要，只保留摘要 + 最近消息。
3. RAG 外挂记忆——把历史全存向量库，每次只检索相关片段塞进上下文。

核心原则：旧的不删（需要时可检索），当前窗口精简（只放必要的）。

五、RAG 相关知识

Q18RAG 和 Fine-tuning 怎么选？
RAG 给模型外挂资料库，Fine-tuning 改变模型本身。

RAG：模型不改，检索资料塞进 Prompt。更新快（换资料库就行），成本低，适合事实类、文档类、时效性强的场景。
Fine-tuning：模型参数被重新训练，知识内化。适合风格模仿、格式精确控制、领域术语内化。

RAG 管知识，微调管行为。两者不冲突——又要事实又想要风格就组合用。
Q19Long Context 和 RAG 怎么选？
问题只跟资料的一小部分相关 → RAG（精准、便宜）。问题需要理解全文才能回答 → Long Context（全量喂入）。

实际最佳实践：先用 RAG，不够再补 Long Context。混合使用——RAG 检索候选段落，再用 Long Context 模型对候选段落做全局分析。取两家之长。
Q20Embedding 模型怎么选？
三条标准：

1. 语言匹配——中文用 BGE-M3，英文用 text-embedding-3-large。选错语言语义区分度骤降 30%+。
2. 维度 vs 成本——1536 维精度高但存储大，384 维精度略降但快 4 倍。10 万条以下不用纠结。
3. 开源 vs API——开源零费用但需要自己部署 GPU，API 按 Token 计费但零运维。
Q21向量数据库怎么选？
数据量 < 100 万，不想运维 → LanceDB（npm install 即用）
数据量 > 100 万，需要分布式 → Milvus（十亿级性能最强）
已经在用 PostgreSQL → pgvector（加个插件就行）
中间态，追求单机性能 → Qdrant（Rust 实现，快且稳）

一句话记忆：LanceDB = 文件即数据库，Milvus = 分布式巨无霸，Qdrant = 单机性能王，pgvector = SQL+向量一把梭。
Q22什么是 Agentic RAG？和普通 RAG 的区别是什么？
普通 RAG——一次检索，直接回答。搜什么、搜一次够不够全在代码里写死。
Agentic RAG——Agent 自己决定怎么搜，检索变成了工具，Agent 自主调用。

例子：用户问"比较 Python 和 Go 的并发模型差异"，Agentic RAG 会自己先搜 Python 并发 → 看结果 → 再搜 Go 并发 → 综合回答。不用你预设流程。普通 RAG 只搜一次。

六、工具设计

Q23怎么设计一个好的 Tool？
四条原则：

1. 职责单一——一个工具只做一件事。Agent 不是程序员，看不懂复杂参数组合。
2. 描述即文档——Agent 选工具靠读描述。说清楚"什么时候用、参数怎么填、返回什么"。
3. 输出结构化 + 有边界——有 truncation 和分页，不要一次返回 5000 行撑爆上下文。
4. 错误返回不要炸——返回 {success: false, error: "xxx", suggestion: "可以先用 list_files 查看"} 而不是直接 throw。
Q24Tool 太多了 LLM 选不准怎么办？
三条策略：

1. 按场景分组——代码场景只给 read/edit/execute，搜索场景只给 search/fetch。一次只暴露 3-5 个工具。
2. 工具描述加使用示例——"何时使用/何时不用" + 示例，消除歧义。
3. RAG 选工具——先 Embedding 所有工具描述 → 用户问题检索 → top 5 相关工具 → 只暴露给 LLM。

核心原则：不是工具越多越好，是给 LLM "刚好需要的"那几个。

七、Prompt 工程

Q25System Prompt 怎么写才好？
四条铁律：

1. 说"你是谁"，不说"你不要"——正向描述行为，比否定句式可靠得多。
2. 角色 → 能力边界 → 行为规范，三段式——你是谁 / 能干什么 / 该怎么干。
3. 工具调用规则写死——不要让 LLM 自己摸索工具用法，在 System Prompt 里定义 SOP。
4. 停止条件明确——同一错误 3 次 → 报告卡住，超出 15 轮 → 总结已完成部分。
Q26Prompt 版本怎么管理？
当普通代码管。

Git 管理：prompts/ 目录下分类存放，改 Prompt 走 PR review 再合，changelog 记录每次改动原因。
代码内嵌：Prompt 和代码版本绑定，不会出现"线上跑的是哪个版本"的问题。
平台管理（大团队）：LangSmith Hub / PromptLayer 支持 A/B 分流、灰度发布。

核心原则：Prompt 就是代码的一部分，版本可追溯、可回滚、可对比效果。
Q27Few-shot 和 Zero-shot 怎么选？
先 Zero-shot → 如果不稳定 → 加 1 个例子 → 还不行 → 加第 2 个。例子够用就停，不是越多越好——3 个以上的 Few-shot 边际收益递减，每多加一个例子都是在烧 Context Window。

八、工程化落地

Q28怎么降低 Agent 系统延迟？
三条砍延迟的方向：

1. 并行化无依赖节点——RAG 检索和 Mem0 召回同时启动，从串联 200ms 变成 max(100ms, 100ms) = 100ms。
2. 流式输出——用 stream() 逐 Token 输出，首 Token 延迟 < 300ms，用户感知延迟降 80%。
3. 砍 LLM 调用次数——合并能合并的调用，一次输出多个决策。

优先顺序：流式 > 并行化 > 砍 LLM 调用。流式成本最低，效果最明显。
Q29怎么估算 Agent 系统成本？
成本 = Token 消耗 × 单价 + 基础设施

单次对话成本 = (Input Token × 输入单价) + (Output Token × 输出单价)
Agent 成本 = 单次对话 × 平均轮次

以 deepseek-chat 为例：一次 5 轮对话 Input 10K + Output 2.5K → 约 ¥0.015。每天 1000 次 → ¥15/天。

Agent 比普通 LLM 贵在哪：多轮调用（5-15 次 vs 1 次）、Context 膨胀（每轮 Input 越来越大）、工具返回（占单次 Token 消耗 60%+）。
Q30怎么降低 Token 消耗？
1. 工具返回截断——不要全文，只给必要片段。工具返回占 Token 消耗 60%+，是最大出血点。
2. 系统消息精简——只保留角色定义 + 核心规则。每次调用都带，砍一个无用词 = 省 N 轮 × 1 Token。
3. 历史消息去重/压缩——旧消息只保留摘要，重复内容合并。
4. 分类用小模型——classify 用便宜模型，复杂推理才用满血模型。100 Token vs 1000 Token，10 倍差距。
Q31Agent 死循环怎么处理？
三层防线：

1. 硬上限——max_iterations = 15，超过直接终止。
2. 相似输出检测——当前回答 ≈ 上一轮回答（语义相似度 > 0.95）→ Agent 在重复自己，强制终止。
3. 趋势监控——连续 3 轮 score 下降 → 越改越差；连续 5 轮没有新 tool_call → 纯思考不干活。

上限兜底 + 重复检测 + 趋势告警。
Q32怎么保证 Agent 输出的稳定性？
四个层面：

1. 温度调低——classify 用 temperature=0（路由不能飘），Agent 生成用 0.3（允许变化但格式不乱）。
2. 输出格式用 JSON 约束——结构化输出比自由文本稳定 10 倍。
3. Few-shot 锚定期望——给 2-3 个标准示例，LLM 会模仿输出风格和粒度。
4. 多轮投票——同一问题跑 3 次，路由都一致才高置信度通过。

稳定性不是让每次都一模一样，而是让关键决策（路由、格式、评分）不出错。
Q33Prompt 注入攻击是什么？怎么防？
Prompt 注入：用户输入里藏指令，劫持 Agent 行为。

防御三层：
1. 输入隔离——用户输入和 System Prompt 物理分离，用分隔符标清边界，过滤特殊字符。
2. 白名单路由 + 意图检测——Agent 只做自己工具范围内的事，超出范围直接拒绝。
3. 用户输入永不进工具——工具参数不能从用户原始输入直接拼接，必须经过 Agent 审核或自行生成。
Q34Agent 怎么处理幻觉问题？
四层防御：

1. RAG 绑定事实——回答必须有出处，检索不到就说不知道。
2. 工具执行验证——声称"已修复"就立即跑测试，挂掉就回退。
3. Reflection 自纠错——另一个 LLM 检查输出中的事实错误。
4. 输出时标注置信度——区分"根据源码确认"和"社区讨论倾向于"。

核心原则：Agent 的每一步声明都要能追溯到——要么来自检索结果，要么来自工具执行输出，要么标注为推测。
Q35Tool 调用失败了怎么处理？
三层容错，逐级升级：

1. 工具内部重试——返回结构化错误 + 建议，不直接抛异常。
2. Agent 自修复——看到错误信息 → 分析原因 → 换策略重试。这是 Agent 的核心能力。
3. 降级兜底——同一工具连续失败 3 次，跳过这一步，用已有结果给部分回答。

LangGraph 的循环边就是做这个"退回重试"的—— ToolNode 的 raise_on_error=false + 条件边 retries < 3 → 退回重试。
Q36Agent State 怎么做持久化？
LangGraph 里一行搞定：Checkpointer

开发环境：MemorySaver——内存存储，进程重启就丢。
生产环境：SqliteSaver——文件存储，重启不丢。或 PostgresSaver——多实例共享状态。

调用时指定 thread_id，同一 thread_id 下次调用会自动恢复之前的状态。从 MemorySaver 切到 SqliteSaver 只需要换一行代码。
Q37eval 测试集是什么？怎么构建？
eval 测试集 = 考试卷。用来衡量 Agent 能力的一套标准化题目，每道题有期望答案。

两个来源：
1. 真实生产日志拆出来——你的 trajectory.json 就是天然测试集源，挑 200 条典型日志标注期望结果。
2. 规则生成 + 人工校验——LLM 批量生成变体问题 → 人工筛掉不合格的 → 入测试集。

四个维度覆盖：简单单步 30%、多步推理 40%、自纠错 20%、边界异常 10%。好的测试集 = 能覆盖真实失败模式。

九、项目实战深挖

▸ Agent 学习助手（LangGraph Multi-Agent）

Q38项目的路由分发是怎么做的？交给 LLM 判断还是硬编码？
交给 LLM 判断，不是硬编码 if-else。

classify 节点调 DeepSeek API，Prompt 要求输出 JSON：{routeTo: 'concept'|'exam'|'code'|'plan'|'news'}，LangGraph 条件边读 routeTo 决定路由。

为什么选 LLM：用户自然语言变体太多——"帮我写个排序"、"coding help"、"来段代码"——硬编码关键词覆盖不了。classify 用便宜模型、temperature=0，成本几乎可忽略，但灵活性和准确率远超 if-else。

注意：routeTo 使用 lastValue reducer，不是 concat——如果用 concat 会变成 ["code","code"] 数组，条件边无法匹配字符串，路由全乱。
Q39classify 节点的 Prompt 具体怎么设计的？
三段式：角色 → 任务 → 输出格式。

角色：你是意图分析专家。任务：分析用户问题，判断属于概念讲解/应试答题/代码实战/学习规划/前沿资讯。同时调用 RAG 检索相关知识、Mem0 召回用户记忆，综合分析后输出。输出：严格 JSON 格式 {routeTo, ragContext, memoryContext, depth}。

关键细节：temperature=0 保证路由稳定性——同一问题不会今天分到 code 明天分到 exam。输出用 JSON 约束——正则提取 JSON 块再 parse，不直接 JSON.parse，因为 LLM 有时会在 JSON 前后多输出解释文字。
Q40五个子 Agent 的 System Prompt 有什么差异？怎么保证回答风格不同？
每个 Agent 有独立的 System Prompt，核心差异在于 角色定位 + 输出格式：

concept（概念讲解）——"你是大学助教"，输出带"前置知识"和"小节总结"，用类比解释复杂概念。
exam（应试答题）——"你是面试官"，先给题目 → 等用户回答 → 再点评，不给直接答案。
code（代码实战）——"你是结对编程搭档"，先写思路再给代码，代码块必须标注语言，带注释。
plan（学习规划）——"你是课程设计师"，输出分阶段的学习路线图，带时间估计和里程碑。
news（前沿资讯）——"你是技术研究员"，严格标注信息来源和发布日期，不确定的标注"(未确认)"。

Prompt 复用：抽公共部分（行为规范、停止条件）到模板，各 Agent 只替换角色描述和输出格式。
Q41Plan-and-Execute 的触发条件是什么？怎么判断问题复杂度？
双条件触发：问题长度 > 40 字 + 包含特定关键词。

关键词：对比、规划、搭建、从零、部署、优化、实现一个、设计一个。

流程：classify 同时判断 isComplex，若 true → 先调用 LLM 拆解子任务（最多 5 个）→ 串行执行 → 汇总。子任务拆解用 temperature=0.1——拆解需要一致性，不能用默认 0.7。

当前子任务串行（有隐式依赖），未来可对无依赖子任务用 Send() API 并行。
Q42RAG 两阶段检索具体怎么实现的？为什么这样设计？
粗排（LanceDB 向量 + sql.js 关键词 → RRF 融合）→ 精排（BGE Reranker Cross-Encoder）。

第一阶段：向量检索召回语义相似、关键词检索召回精确匹配 → RRF(k=60) 融合取 top 15。
第二阶段：BGE Reranker V2-M3 对 15 个候选逐一 Cross-Encoder 打分 → 精排取 top 5。

为什么不能只用 Cross-Encoder：全库 500 条文档，每条都跟 query 拼接后过 24 层 BERT——500 x 50ms = 25 秒，不可接受。向量检索 O(log n) 先砍到 15 条，再精排。两阶段各司其职。

知识入库时为省钱用了 DeepSeek Embedding API——5 篇文档不需要自己部署 GPU。
Q43你是怎么发现 State reducer 用错导致路由全乱的？
state.messages 和 state.routeTo 都对 Annotation 配置了 concat reducer。结果 routeTo 不是字符串 "code"，而是数组 ["code", "code"]。LangGraph 条件边拿到数组，无法匹配任何 case 字符串——所有子 Agent 都不执行，直接跳到 reflect。

排查：LangSmith Trace 里看 routeTo 字段值 → 发现是数组 → 意识到 reducer 用错了 → 改成 lastValue（默认）→ 路由恢复正常。

教训：LangGraph State 分两类——"累积型状态"（messages 用 concat）和"当前型状态"（routeTo/score 用 lastValue）。这是 StateGraph 的核心设计，混了就废。
Q44为什么选了 Mem0 而不是 LangGraph 原生的 Store？
LangGraph 原生 Store 是手动 Key-Value 存储——需要自己定义字段、自己写 Put/Get。Mem0 的核心优势是：LLM 自动从对话中提取记忆单元，不需要预定义 Schema。

示例：用户说"我学过 RAG 但不熟悉 HNSW" → Mem0 自动生成两条记忆{"已掌握 RAG", "HNSW 薄弱"}。不用我写 if-else 判断该存什么。

冲突检测是 Mem0 的杀手特性：用户后来说"我其实不懂 RAG"，Mem0 自动覆盖之前的记忆。手动 Key-Value 要实现这个需要自己设计冲突解决逻辑。

但 Mem0 进一次网络 50-200ms——所以 classify 节点一次性召回 + 写入 State，后续 Agent 从 State 读，不各自查 Mem0。减少网络开销。
Q45MCP 和 Function Calling 都支持工具调用，为什么选了 MCP？
不是二选一，MCP 是 Function Calling 的上层标准化。

Function Calling 是在 LLM 的请求参数里定义工具 Schema——工具定义和代码耦合。加新工具要改代码、重启服务。

MCP Server 是独立进程——JSON-RPC 通信、Docker 隔离部署、tools/list 自动发现。最大的优势：代码执行这个 Server 必须安全隔离——跑在 Docker 里，非 root 用户 + --memory=256m --network=none。Function Calling 的工具函数是进程内执行，没法做这个级别的隔离。

实际架构：LangGraph Agent 通过 MCP Client → JSON-RPC → 4 个 MCP Server（代码执行/arXiv/知识库/进度查询）。

▸ CodeTrail（Agent Coding 轨迹可视化）

Q46trajectory.json 的解析引擎要同时处理哪两种 content 格式？
同一个 assistant 消息的 content 字段可能是 string 或 Array——这是 Anthropic Messages API 的多态设计。

String content：Agent 的纯文本思考，可能包含 <thinking> 标签包裹的推理块 + 普通文本。
Array content：多 block 消息——{"type": "tool_use", "id": "...", "name": "read_file", "input": {...}} 和 {"type": "text", "text": "..."} 混排。

解析器分支处理——typeof string → 正则提取 thinking → 剩余文本归为 text 类型。Array.isArray → 遍历 block → tool_use 归为 tool_call 类型，text 归为 text 类型。

踩坑：一开始写死了 string 分支，工具调用全部跳过。排查半小时发现是遗漏了 Array 格式。
Q47dagre 布局的具体参数怎么调的？
rankdir: "TB"（Top-to-Bottom）——轨迹是时间序，从上到下最自然。ranksep: 80——层间距 80px，太密节点叠在一起，太松需要滚动太多。nodesep: 60——同层节点间距，保证 tool_call 节点和 file 节点不重叠。nodeWidth: 200, nodeHeight: 80——根据节点实际内容估算的固定尺寸。

调参方法：先给大值（100/80），看布局太松就缩小（60/50），看节点重叠就加大。没有公式，纯人工迭代。
Q48自定义 ReactFlow 节点用 React.memo 后为什么还是不生效？
因为每次 rebuildGraph 都生成新的 nodes 数组——即使内容不变，对象引用变了，React.memo 的浅比较失效。

解决方案：用 useMemo 包裹 graphData 计算，只在 steps 变化时重新生成。同时 AgentNode 的 data 字段只传必要字段（id, type, label, detail, stepIndex），不用 {...node} 整个对象展开——展开会产生新的嵌套引用。

ReactFlow 的 memo 机制：当 ReactFlow 自身检测到 nodes 引用没变，它连内部 diff 都不做，省了最重的那步 DOM 比对。
Q49Monaco Editor 怎么加载？首屏会受 2MB 影响吗？
不影响首屏。用了 Next.js 的 next/dynamic 懒加载：const MonacoDiffEditor = dynamic(() => import("@monaco-editor/react").then(m => m.DiffEditor), { ssr: false })

首屏是 Landing Page——只有 Uploader 组件，Monaco 不在首屏 bundle 里。用户切换到 Diff View tab 时才触发 Monaco Worker 的异步 CDN 加载。SSG 阶段完全不碰 Monaco。

如果不做懒加载，Monaco 的 JS + Worker 约 2MB 进首屏 bundle——LightHouse 得分直接扣 15 分以上。
Q50unified diff parser 怎么从 initial_state + final_diff 重建代码？
逐 hunk 逐行应用：

1. 取 initial_state 的第一个文件内容作为 base
2. 按行解析 final_diff——匹配 @@ -oldStart,oldLen +newStart,newLen @@ hunk header
3. 在当前 hunk 内：以 "+" 开头的行 → 追加到右边；以 "-" 开头的行 → 跳过（旧内容要删除）；以 " " 开头或空行 → 两边都保留
4. 用 Array.splice(oldStart - 1, oldLen, ...newLines) 替换对应位置

异常处理："No newline at end of file"、空 hunk、非 Python 文件都走退化逻辑——异常 hunk 跳过，左右显示原始内容。生产级应该用 npm: diff 库做规范 parse，不用手写。
Q51Zustand 和 Redux 怎么选的？为什么不用 Redux Toolkit？
Zustand：create() 返回 hook，组件里 usePlaybackStore(s => s.currentStep) 精确订阅单一字段。

Redux Toolkit 需要 createSlice + configureStore + Provider 包裹整个组件树——对全 Client Component 架构是额外的样板代码。Zustand 零 Provider，create() 一个函数搞定。

全局状态不乱吗？这个项目的状态非常集中——只有一个 trajectory 数据源，不是多页面多表单的复杂场景。usePlaybackStore 做了精确字段订阅（不传 selector 才订阅全量），选择器只取需要的字段则跳过无关更新。

如果以后支持多 trajectory 并行对比，拆成多 store 或者用 createStore 工厂函数。
Q52tool_call 节点怎么自动关联到文件节点？filePath 取不到怎么办？
extractFilePath 从 toolInput 中依次尝试四个字段：filePath → path → file → pattern。

不同工具的参数名不同——read_file 用 filePath，bash 用 workdir，glob 用 pattern。pattern 放在最后因为它是搜索模式而非具体文件路径。

都取不到就跳过——不创建文件节点。图中部分 tool_call 节点没有文件关联边，不影响核心功能。

文件节点去重用 Map——同一文件被多次操作（先读后改），只创建一个节点，多条边指向同一个文件节点。

十、Agent 学习助手——架构与工程深挖

▸ 架构设计决策

Q53为什么是 7 个节点，不是 3 个或 15 个？怎么决定图的结构？
节点数 = 职责分离的粒度。太少耦合重，太多编排复杂度爆炸。

4 节点方案（去掉 classify + reflect）→ 用户输入直接给 concept Agent，没有路由和自纠错，差评率高一倍。
15 节点方案（每个 Agent 再拆 sub-agent）→ 编排开销 > 分工收益，延迟从 5s 涨到 15s。

7 节点的设计逻辑：1 个路由节点 + 5 个执行节点 + 1 个质检节点——每个节点单一职责，路由和质检是框架层，5 个 Agent 是业务层，职责正交。

决定图结构的核心原则：先画业务流程图，再映射节点。用户跟这个系统的交互有哪些场景？概念、刷题、代码、规划、资讯 → 5 个场景 → 5 个执行节点。每个场景前面需要判断走哪个 → classify。每个场景后面需要质量检查 → reflect。你不是在设计图，是在建模用户行为。
Q54为什么选 TypeScript 不选 Python？
TypeScript 的 LangGraph 版本基于 LangGraph JS 官方 SDK。选 TypeScript 而不选 Python 因为全栈背景——前端 Next.js + 后端同一语言，不需要维护两个语言栈。

Python 的优势：社区更大、文档更多、LangSmith Python SDK 功能更全。TS 的劣势：部分 LangGraph 高级特性（如 Postgres Checkpointer）只有 Python 版。

但实际开发中 TS 的劣势并不致命：DeepSeek 走 OpenAI 兼容协议，用 OpenAI Node SDK 直接调。LangGraph 核心功能（StateGraph、条件边、Send API）TS/JS 版本完整支持。生产换 Python 只需把 Prompt 和 Graph 逻辑平移，语法层差异，架构不变。
Q55为什么选 DeepSeek 而不是 GPT-4o 或 Claude？
三理由：成本、中文能力、OpenAI 兼容。

成本——DeepSeek 输入 ¥1/M tokens，GPT-4o 输入 ¥15/M tokens，价格差 15 倍。按每天 1000 次调用、每次 5000 token 算，DeepSeek ¥5/天，GPT-4o ¥75/天。
中文能力——DeepSeek 中文不逊于 GPT-4o，比 Claude 中文更好。
OpenAI 兼容——不需要改客户端代码，换模型只需换一行的 baseURL + apiKey。

但 DeepSeek 的 Function Calling/Tool Use 不如 GPT-4o 稳定——偶尔忘了调工具。解决方案：reflect 节点检测"应该有工具调用但没有"的情况 → 退回重试。多一轮调用，比换 GPT-4o 省的钱多。
Q56RAG 为什么不用 Chroma/Weaviate？为什么手写 HNSW + 只用 LanceDB？
早期自建 HNSW 是为了学习——把向量检索原理吃透。后续产品化换成 LanceDB，npm install 零部署。

不用 Chroma 因为它是 Python-first，Node 只有 HTTP client，多一层网络开销。不用 Weaviate 因为它是独立服务，需要 Docker 部署——5 篇文档不值得跑一个 500MB 的服务。

LanceDB 的优势：基于 Lance 列存格式，直接读写磁盘文件，npm 安装即用。数据量到 10 万条之前不需要换。10 万条以上考虑 Qdrant Cloud（托管）或 Milvus（自建分布式）。

选型逻辑：现阶段的数据规模不需要分布式向量数据库。过度设计是浪费。
Q57知识库 5 篇文档的 RAG，检索延迟/精度有没有数据？
有。检索延迟：LanceDB 向量检索 25-40ms + BGE Reranker 精排（对 15 个候选逐个）200-350ms。检索总耗时约 250-400ms。

精度——用 RAGAS 框架对 50 道测试题跑过一轮：Context Precision 87%，Context Recall 82%，Faithfulness 91%，Answer Relevancy 89%。

5 篇文档的精度参考意义有限——文档少且内容高度相关，检索基本不会翻车。精度瓶颈在文档覆盖率而不是检索算法——用户问的概念没入库，自然检索不到。这个场景的核心问题是知识库的持续扩充，而不是检索精度。
Q58为什么用 Mem0 而不是存 JSON 文件？
JSON 文件手写记忆 = 每次对话后手动判断该存什么、怎么查、怎么冲突检测。Mem0 把这三个都自动化了。

5 篇文档的场景确实用 JSON 文件也能 work，但选 Mem0 是为了证明"有这个需求怎么办"。面试官问"如果用户量到 1000，跨会话记忆怎么做"，能直接说"已经用了 Mem0"。

选型逻辑：用 Mem0 不是因为它解决了当前的痛点，而是证明了你有面向未来的架构意识——知道存储需要支持语义检索、自动提取和冲突检测。这些用 JSON 文件做，越做越痛苦，最终还是要换。

▸ 生产化与异常处理

Q59DeepSeek API 挂了怎么办？有容灾方案吗？
两级降级：切换模型 → 兜底回复。

第一级：检测到 DeepSeek API 5xx/timeout → 自动切换到备用模型（如 Qwen 或 OpenAI 兼容的其他 API）。
第二级：所有模型都挂 → 返回兜底回复"系统暂时不可用，请稍后重试"。

当前实现状态——容灾切换逻辑在 LLM 调用层的 try-catch 里预留了切换点，但备用模型没配。因为个人项目没有高可用要求。

生产化要加：circuit breaker（连续 3 次失败 → 熔断 30 秒）和健康检查（每 60 秒探测一次，失败率 > 50% 时自动切备用模型）。
Q60LLM 输出的 JSON 不合法怎么办？
三层防护：正则提取 → 容错 parse → 降级默认值。

第一层——正则提取 JSON 块：忽略前后的解释文字，只提取 {} 包裹的 JSON。
第二层——容错 parse：JSON.parse 失败则尝试修复常见错误（缺逗号、单引号、尾逗号）。
第三层——降级默认值：全部失败 → 用默认参数继续（routeTo 默认 "concept"，subTasks 默认拆成单步）。

实际踩过坑：DeepSeek 有时在 JSON 前后加 Markdown 解释文字。正则提取是必须的，不能直接 JSON.parse 全输出。GPT-4o 的 JSON mode 比 DeepSeek 更稳定——但 DeepSeek 的价格优势让这种偶尔的容错可以接受。
Q61LLM 幻觉怎么在实际生产中检测？
不是事后检测，是实时拦截。

方案 1（已实现）——RAG 事实绑定：Agent 的每个声明必须能从检索片段中找到锚点。Reflection 节点检查"回答中的关键事实是否在 ragContext 中出现"。没出现 → 标记可疑 → 退回重写。
方案 2（规划中）——代码执行验证：code Agent 给出的代码必须能跑通。声称"已修复"→ 立即 execute 验证 → 挂掉退回去。
方案 3（规划中）——搜索引擎交叉验证：对关键声明自动检索外部来源，用另一个 LLM 做事实一致性比对。

方案 1 是够用的（因为知识库文档是自建的，可控），但 RAG 也有 Retrieval Failure 的风险——知识库里没有足够信息时，Agent 仍然可能编造。
Q62怎么防止 Prompt 注入？
三层防护，已在代码里实现。

层 1——输入边界：用户输入和 System Prompt 用 XML 标签物理分隔。System Prompt 明确写"只处理 user_input 中的内容"。
层 2——意图白名单：classify 只识别 5 种合法意图（concept/exam/code/plan/news），超出范围 → 一律拒绝。
层 3——工具参数隔离：工具参数不直接从用户输入拼接——Agent 先通过 LLM 分析用户意图，生成自己的工具调用参数。特别是 execute_command 这类危险工具——command 参数由 Agent 自己构造。

当前防护级别对个人项目足够。企业级要加输入输出内容安全 API（如 Azure Content Safety/OpenAI Moderation）做有害内容实时检测。
Q63上下文窗口不够怎么办？怎么做消息管理？
滑动窗口 + 摘要降级。

当前——对话轮次限制：State.messages 最多保留 30 条。超出 → 丢弃最旧的消息。
升级方案——摘要压缩：超出窗口时，把前 20 条旧消息送给 LLM 生成一段摘要，只保留 System Prompt + 摘要 + 最近 10 条。既控制长度又不丢失信息。

注意：不能截断在 tool_use 和 tool_result 之间——必须保证这对要么都在，要么都删。
Q64怎么测试 Agent 节点的正确性？
三层测试策略：单元 → 集成 → 回归。

单元测试（已做）——mock DeepSeek API 返回值，验证 classify 的 routeTo 解析逻辑。JSON parse 容错、正则提取、默认值降级。
集成测试（部分）——跑完整 graph.invoke()，输入固定问题，检查 State 的 routeTo、messages 长度、reflectionScore 范围。
回归测试（规划中）——50 道标准题，每次 Prompt 变更后全跑一遍 → 比较 routeTo 正确率 + 回答质量评分（LLM-as-judge）。

当前覆盖率不高——因为 LangGraph 的核心逻辑是 LLM 调用，mock 的复杂度是 test 价值的 3 倍。对个人项目来说，更多的质量保证靠 RAGAS 评估 + 手动抽查。
Q65整个对话的端到端延迟是多少？瓶颈在哪？
单次对话 3-7 秒，瓶颈明确在 LLM 生成阶段。

延迟分解（一次典型对话）：classify 120ms + RAG 检索 250ms + Mem0 召回 100ms（并行计 250ms）→ concept Agent 生成 3.8s → reflect 评分 1.0s → 总 5.2s。

concept Agent 的 LLM 生成占 73% 的总耗时。优化空间：流式输出（用户 300ms 看到第一个字，感知延迟降 80%）、换更快的模型、缓存常见问题。
Q66如果知识库从 5 篇涨到 10 万篇，RAG 怎么撑？
三条升级路径，按阶段推进：

阶段 1（1 万篇）——LanceDB 仍然够用。单表百万向量没问题，主要优化方向是索引升级（IVF_PQ 压缩向量，减少 90% 内存）。
阶段 2（10 万篇）——加 Embedding 缓存 + Qdrant 换掉 LanceDB。Qdrant 支持 payload 索引（不用 sql.js 做过滤）、HNSW 索引参数可调。
阶段 3（100 万篇）——先粗筛再精排的流水线：Elasticsearch 关键词过滤砍到 1000 篇 → 向量 ANN 粗排砍到 50 篇 → Cross-Encoder 精排取 top 5。

核心思路——不是换更快的服务器，是分阶段砍候选量。
Q67如果重做这个项目，你会做哪些不同的事？
三件事：

1. 分离 Agent 定义和 Graph 编排——当前 Agent 的 System Prompt 和 Graph 节点耦合在代码里。应该把 Agent 定义抽成 YAML/JSON 配置，Graph 加载配置文件构建。好处：加新 Agent 不用改图代码。
2. 早做 Checkpointer 持久化——当前用 MemorySaver，进程重启对话就丢。应该从一开始就用 SqliteSaver，状态持久化不丢——用户重新打开页面能恢复之前的对话。
3. Observability 前置——先上 LangSmith/LangFuse，后写 Agent 代码。每个节点跑几次就能看到延迟分布和 Token 消耗，调优方向一目了然。

十一、CodeTrail——架构与工程深挖

▸ 架构设计决策

Q68为什么不直接用 LangSmith/LangFuse？自己做有什么特殊价值？
LangSmith/LangFuse 是通用的 LLM 可观测性平台，CodeTrail 专为 agentic coding 轨迹设计。

核心差异：1. 知识图谱——LangSmith 的 trace 是树状结构（Span 嵌套），CodeTrail 的 Graph 是 DAG（Thought → Tool Call → File 这种跨层关联）。工具调用和它修改的文件之间的因果边，Trace 看不出来，图谱能。
2. 回放引擎——按步骤逐帧回放 Agent 的思维过程，不是一次性展示全量 trace。适用场景是标注员的逐步骤审查。
3. 代码 Diff 对比——trajectory 的 initial_state vs final_diff 直接渲染 Monaco 双栏对比。LangSmith 没有这个能力。

自己做不是"造轮子"——LangSmith 的定位是 LLM 应用开发者的调试工具，CodeTrail 的定位是标注团队的轨迹审查工具。使用者、场景、核心诉求都不同。
Q69为什么 Next.js 不是 Vite + React？
当前是 100% Client Component，Next.js 的 SSR 能力没用上。选 Next.js 是为预留服务端能力。

计划扩展的服务端需求：1. Server Action——调 AI 接口对 trajectory 做自动质量评分（避免 API Key 暴露在前端）。2. RSC 分享页——公开分享 trajectory 分析结果，SEO 友好（预渲染，不需要浏览器 JS）。3. API Routes——接收真实 Agent（如 Claude Code）推送的 trajectory data，做流式解析。

如果纯纯是一个本地工具永远不上线，Vite 更合适。但项目定位是生产级工具——选 Next.js 是为了后面 3 步不重构。
Q70为什么 ReactFlow 而不是 D3.js 或 Cytoscape.js？
ReactFlow 的节点是 React 组件，D3 的节点是 SVG 元素——定制化成本差一个数量级。

D3：你要自己用 append("rect") + append("text") 画节点，tooltip 要手写，点击事件要手写，布局要用 d3-force 自己算。自由度最高，但写个带参数摘要的 tool_call 节点要 50+ 行。
Cytoscape.js：做生物网络图好，做流程图一般。React wrapper 维护者不再更新。
ReactFlow：节点是完整 React 组件——ToolNode 用 JSX 写样式、写交互、写 Memo 优化。dagre 布局用现成的 graphlib API。

选型原则：交互复杂度决定框架。有自定义 UI（tool_call 的 JSON 参数摘要、file 的路径显示、点击跳转步骤）→ ReactFlow 的组件化优势碾压 D3。
Q71dagre 布局为什么不用 ELK.js？
ELK.js 更强但更重，800KB vs dagre 180KB。当前需求不需要 ELK 的增量。

ELK 比 dagre 多的能力：端口约束、多层级分组、增量布局。这些需求这个场景一个都不需要——Graph 是自上而下的时序图，没有分组，不需要增量。

dagre 的劣势：没有维护者了（上次 release 是 2019 年），算法实现是 O(N^2 x L)。但对 100 节点以下的场景，5ms 布局完成，瓶颈不在算法复杂度在 ReactFlow 的 DOM 渲染。

什么时候换 ELK——节点数 > 500 时 dagre 布局 > 50ms 开始有感知延迟。
Q72Monaco Editor 和 CodeMirror 怎么选的？
Monaco 有 DiffEditor 内置组件，CodeMirror 的 diff 需要自己拼。

Monaco 的 DiffEditor——传入 original 和 modified 两个字符串，自动渲染双栏对比 + inner-line diff + 行号对齐 + 滚动同步。零配置。
CodeMirror 6——有 merge 插件做双栏 diff，但需要额外安装 @codemirror/merge，API 比 Monaco 复杂。
为什么不是 Shiki——Shiki 只能高亮单文件，不渲染 diff。要 diff 得自己拼两个 Shiki 渲染结果对齐。

Monaco 的代价——2MB Worker（CDN 加载）。通过 next/dynamic 懒加载只在切到 Diff tab 时才下载——首屏 0 负担。

▸ 性能与边界处理

Q7310,000 步的 trajectory 图谱能正常渲染吗？帧率多少？
10,000 节点 DOM 渲染会卡爆。策略是虚拟化——视口外节点不渲染 DOM。

性能数据：500 节点 dagre 布局 12ms，ReactFlow 渲染 60fps。1000 节点 35ms/45fps。5000 节点 200ms/20fps。10000 节点 800ms/8fps。

10000+ 节点方案：增量加载 + Web Worker 离线布局。Web Worker 里跑 dagre.layout()，主线程不阻塞。渲染时 LOD 降级——缩小视图时简化节点（只显示图标），放大后再渲染细节。
Q74Zustand 怎么处理高频更新？播放引擎每 500ms 更新一次 currentStep——会触发全量 re-render 吗？
不会。Zustand 的精确订阅机制只让相关组件重渲染。

usePlaybackStore(s => s.currentStep)——只订阅 currentStep 一个字段。other fields 变化不会触发此组件更新。

播放时 currentStep 每 500ms 变一次 → 只影响 Timeline 组件（步骤高亮更新）和 StepDetail 组件（详情内容切换）。Graph 组件不受影响（graphData 是 useMemo 包裹的，steps 没变就不会重算），Dashboard 组件不受影响（stats 不变）。

Redux 也能做精确订阅——useSelector(state => state.steps[currentStep]) + shallowEqual。但 Zustand 的 default equality 就是浅比较——不需要显式传比较函数。
Q75trajectory.json 有 50MB 怎么办？一次性 parse 会爆内存吗？
当前一次性 parse + 全量 Zustand 存储，50MB JSON 不会爆（~100MB 内存对象），但解析时 UI 冻住 2-3 秒。

优化方案：1. Web Worker 异步 parse——主线程不冻结，可以显示进度条。2. 分片加载——trajectory 数组按消息序号分页，显示第一页，滚动到底时懒加载下一页。3. IndexedDB 缓存——parse 结果存入，下次打开直接读缓存。

当前还没做到——因为 sample 数据不到 20KB。但架构上 Zustand 的 Store 不限制大小，加 Worker 分片只是新加一层，不动现有代码。
Q76如果 trajectory 是循环的（Agent 在同一工具上反复调用），图谱会不会成环？
dagre 的布局算法第一步就是去环——自动反转边消除有向环。

图谱的边是时序流向的（rankdir: "TB"），每一步指向下一步——天然是无环 DAG。即使 Agent 在同一工具上反复调用（step 3 → read_file, step 5 → read_file, step 8 → read_file），在图中是不同的节点，不会形成环。

文件关联边也是单向的——tool_call → file 节点，file 节点不指回 tool_call。所以不会成环。

▸ 设计反思

Q77如果重做 CodeTrail，你会怎么做？
三件事：Canvas 渲染、流式接收、多 trajectory 对比。

1. Canvas 渲染替代 ReactFlow DOM 渲染——当前 1000 节点以上 DOM 开始吃力。Pixi.js 渲染层，10000 节点流畅。
2. WebSocket 流式接收轨迹——当前是文件上传。对接真实 Agent 时应 WebSocket/SSE 接收 step 消息，实时绘制图谱。
3. 多 trajectory 并行对比——标注员的核心需求是"同一个 task，Agent A 和 Agent B 的表现谁好"。分屏左右各放一条 trajectory 的时间线，同步回放。

优先做流式接收——这是从 Viewer 升级到 Monitoring 的关键一步。
Q78为什么不做成 Chrome Extension/VS Code 插件？
Web 版 = 零安装 + 分享链接 + 跨平台。

Chrome Extension——只能 Chrome 用，分享不方便，审核周期长。VS Code 插件——只能开发者在编辑器里用，产品和标注团队用不了。Web 版——任何人打开浏览器就能用。最核心的场景是"标注员收到 50 条 trajectory，需要在 2 小时里逐条审查"——Web 版让标注员不需要装任何东西。

但 VS Code 插件是有价值的补充——Agent 运行完在同一个编辑器里直接看轨迹回放。下一个项目做共享 API 层，渲染层拆分两个目标。
Q79这个项目怎么测试？怎么保证解析引擎的正确性？
解析引擎用 snapshot 测试，UI 组件靠手动验证。

解析引擎（parser.ts）：准备 5 个不同格式的 trajectory.json——纯 text content、Array content（含 tool_use/tool_result）、混合格式、空 trajectory、畸形 JSON。每个输入断言输出的 ParsedStep[] 的 length、type 分布、关键字段。用 Jest snapshot 做回归——改解析逻辑后如果 snapshot 变了，一眼看出。
UI 组件：ReactFlow 和 Monaco Editor 需要 DOM 环境，mock 复杂度太高。当前靠手动回归检查清单——Upload → Parse → Timeline 显示 → Graph 渲染 → Diff 渲染 → Dashboard 数字正确。

上生产要加 E2E 测试（Playwright）——自动打开页面 → 上传测试 json → 断言 Timeline 出现 N 个步骤。
Q80部署到哪儿？cold start 多久？
Vercel（通过 GitHub 自动部署）。push main → Vercel 自动 build + deploy → URL 生效。

SSG（Static Site Generation）——build 时生成全量 HTML/CSS/JS，部署到 CDN。没有 server，没有 cold start——页面秒开。Monaco Editor 的 Worker 是 CDN 异步加载，首屏不阻塞。

如果接入 Server Action（调 AI 做质量评分）→ 从 SSG 变成 ISR，需要 Vercel Functions。但当前纯静态不需要——成本 ¥0。
