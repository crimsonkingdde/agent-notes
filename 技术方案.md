# Agent 学习助手 — 技术方案与架构文档

## 项目概述

面向 Agent/大模型工程领域的交互式学习系统。TypeScript + Node.js 全栈开发，集成 DeepSeek API 作为推理底座，基于 LangGraph 构建 Multi-Agent 编排引擎。

---

## 一、已实现的核心架构

### 1.1 整体数据流

```
用户输入 → classify 节点(意图+RAG+Mem0) → 条件边路由 → 5个子Agent节点 → reflect 节点 → 返回
                                    ↑                                              │
                                    └──────────── 循环边(needsRetry) ──────────────┘
```

### 1.2 技术选型（实际使用）

| 层级 | 选型 | 替代方案 |
|------|------|---------|
| 语言/运行时 | TypeScript + Node.js | Python (FastAPI) |
| LLM 底座 | DeepSeek API (OpenAI 兼容) | GPT-4o / Claude |
| Agent 编排 | LangGraph StateGraph | 手写循环 / AutoGen / CrewAI |
| 向量存储 | LanceDB (初期自建 HNSW) | Qdrant / FAISS |
| 元数据存储 | sql.js (纯 JS SQLite) | PostgreSQL |
| 记忆引擎 | Mem0 | Zep / 手写 SQL |
| Reranker | BGE Reranker V2-M3 | Cohere Rerank |
| 工具调用 | MCP 协议 (JSON-RPC) | Function Calling |

### 1.3 State 设计

```typescript
const AgentState = Annotation.Root({
  // 消息历史（reducer: concat）
  messages: Annotation<BaseMessage[]>({
    reducer: (c, u) => c.concat(u), default: () => []
  }),
  // 控制字段（lastValue 覆盖）
  routeTo: Annotation<string>(),
  depth: Annotation<string>(),        // beginner | advanced | research
  mode: Annotation<string>(),          // tutor | exam
  reflectionScore: Annotation<number>(),
  reflectionRetries: Annotation<number>(),
  needsRetry: Annotation<boolean>(),
  // RAG 和记忆上下文
  ragContext: Annotation<string>(),
  memoryContext: Annotation<string>(),
  // Plan-and-Solve
  subTasks: Annotation<SubTask[]>(),
  isComplex: Annotation<boolean>(),
});
```

### 1.4 图结构

```
START → classify → [条件边: routeTo]
                      ├─ concept → reflect
                      ├─ exam    → reflect
                      ├─ code    → reflect
                      ├─ plan    → reflect
                      └─ news    → reflect
                                    │
                          [条件边: needsRetry?]
                           ├─ true  → 退回 Agent (循环)
                           └─ false → END
```

## 二、RAG 检索增强（已实现）

### 2.1 离线入库

```
知识条目 → 递归字符切分(300tk/块) → 父子文档索引(子300tk/父2000tk)
  → DeepSeek API Embedding (1536维) → LanceDB 向量库 + sql.js 元数据
```

### 2.2 在线检索（两阶段）

```
用户问题 → Embedding → LanceDB向量检索 + sql.js关键词检索
  → RRF 倒数排名融合(粗排 top 15)
  → BGE Reranker V2-M3 Cross-Encoder 精排(细排 top 5)
  → 父文档展开(子文档→parent_id→完整父文档)
  → 注入 LLM Prompt
```

### 2.3 检索策略细节

- **递归切分：** 优先按 `\n\n` → `\n` → `。` 边界切，保持语义完整
- **父子文档：** 小块建索引保精度，大块返回保上下文完整
- **RRF 融合：** k=60 平滑常数，自动平衡向量和关键词的贡献
- **BGE Reranker：** 24层 BERT Cross-Encoder，拼接编码做跨句注意力

## 三、Agent 推理范式（已实现）

### 3.1 Plan-and-Solve

```
触发条件: 问题 > 40字 + 包含(对比|规划|搭建|从零等关键词)
流程: LLM分解→JSON子任务列表→逐一执行→结果合成
特点: 子任务串行（有隐式依赖），最多5个子任务
```

### 3.2 Reflection 自纠错

```
四维评分: 准确率(0.35) + 清晰度(0.30) + 可运行性(0.15) + 满意度(0.20)
评估: LLM 输出 JSON {score, issues, shouldRetry, retryStrategy}
修正: score < 0.75 → 诊断问题 → 修正重答 → 最多3轮
实现: LangGraph 循环边 — 条件边函数控制 needsRetry && retries < 3
```

## 四、记忆系统（已实现）

### 4.1 四层架构

| 层级 | 存储 | 生命周期 | 作用 |
|------|------|---------|------|
| 工作记忆 | Map + TTL 300s | 单次推理 | 临时草稿 |
| 会话记忆 | 内存 Map | 本轮对话 | 追问上下文 |
| 长期记忆 | Mem0 | 永久 | 跨会话知识追踪 |
| 画像记忆 | Mem0 | 永久 | 水平/薄弱点/偏好 |

### 4.2 Mem0 vs LangGraph Store

选择 Mem0 而非 LangGraph 原生 Store：
- Mem0 自动从对话中提取记忆单元（不需要手动定义字段）
- 内置冲突检测（"我完全不懂 RAG" → 自动更新之前"已掌握 RAG"）
- 语义检索而不是手动 Key-Value

## 六、生产化改造（架构设计已完成，逐步实现中）

### 6.1 流式输出

**当前：** invoke 同步等全文 → 控制台一次性打印
**设计：** stream_events() → Token 级别实时输出

```typescript
// 替换同步 invoke
for await (const event of graph.streamEvents(input, { version: "v2" })) {
  if (event.event === "on_chat_model_stream") {
    process.stdout.write(event.data.chunk.content); // 逐字打印
  } else if (event.event === "on_tool_start") {
    console.log(`\n🔧 调用工具: ${event.name}`);
  }
}
```

前端对接：Next.js + Vercel AI SDK → `streamText()` → `useChat()` 自动接 SSE。不再手写 HTML EventSource。

### 6.2 Context 分离（Runtime 三层架构）

**当前：** user_id、配置参数混在 AgentState 和 CONFIG 全局常量中
**设计：** LangGraph Runtime 三层分离

```
State  ← 放请求级动态数据（messages, routeTo, reflectionScore）
         reducer 控制合并策略（concat / lastValue）

Store  ← 放跨会话持久化数据（用户画像、学习记录、经验记忆）
         Mem0 实现，按需读写

Context← 放运行时静态配置（user_id, max_tool_calls, model）
         graph.invoke() 时通过 configurable 注入
         节点内通过 config.configurable 读取
         只读——不会被修改、不会被序列化、不会被注入 LLM Prompt
```

调用方式：
```typescript
await graph.invoke(
  { messages: [...] },
  { configurable: { thread_id: "session_123", user_id: "u_001", max_tool_calls: 10 } }
);
```

### 6.3 完整 ReAct 循环

**当前：** tryToolCalls —— 只调一次工具就结束。LLM 看到工具结果后不能再次决定"还需要更多工具"
**设计：** LangGraph 条件边 + 循环边实现标准 ReAct

```
agent → [条件边: hasToolCalls?]
  ├─ true  → tools → agent (循环——LLM 看到结果后重新决策)
  └─ false → reflect → END

每轮循环:
  1. LLM 收到完整上下文(messages + tool_history + observations)
  2. LLM 决定: 继续调工具 或 Final Answer
  3. 如果继续 → 执行工具 → 结果追加 → 回到步骤1
  4. 如果结束 → 进入 reflect 校验

上限: 15 轮（防止死循环）
和 LangGraph 的 AgentExecutor 区别: AgentExecutor 是黑盒循环，手写是白盒——每一步消息拼接可控
```

### 6.4 并行执行（Fan-out / Fan-in）

**当前：** Plan-and-Solve 子任务串行执行
**设计：** Send API 实现无依赖子任务并行

```typescript
// 替换串行 for 循环
function routeSubTasks(state: AgentState): Send[] {
  return state.subTasks.map(task =>
    new Send(task.agentName, { query: task.description })
  );
}
// LangGraph 自动并行执行所有 Send 目标 → Fan-in 时等待全部完成 → 汇总
```

延迟：3个独立子任务从 3×2s=6s 降到 max(2s)=2s

### 6.5 图级错误处理（三层防线）

```
第一层: 工具函数内部 try-catch + 参数校验（即时）
第二层: ToolNode raise_on_error=false + 自定义重试（秒级）
第三层: 条件边错误恢复路由（流程级）
        重试 < 3次 → 退回 agent 重试
        重试 ≥ 3次 → 降级回答（基于已有结果给部分答案）
        严重错误 → 终止并告知用户
```

### 6.6 可观测性

**当前：** Date.now() 算总耗时，无分步追踪
**设计：** LangSmith Tracing + RAGAS 评估

```
LangSmith Trace:
  classify 节点: 120ms, 50 tokens
  RAG 检索: 45ms, 5 results
  concept 节点: 3.8s, 500 tokens
  reflect 节点: 1.0s, 200 tokens
  Total: 5.2s, 930 tokens ← 瓶颈一目了然

RAGAS 指标:
  Context Precision / Recall / Faithfulness / Relevancy
  200 道测试题，每周跑一次，监控趋势
```

### 6.7 Human-in-the-loop 断点

**设计：** interrupt() + Command 恢复机制，应用于代码执行确认

```typescript
// 代码执行前设断点
function beforeCodeExecute(state: AgentState) {
  const review = interrupt({
    type: "code_review",
    code: state.agentOutput,
    question: "是否确认执行此代码？"
  });
  return review === "approved" ? execute(state) : { cancelled: true };
}
// 用户确认后通过 Command(resume="approved") 恢复执行
```

## 七、技术决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| Agent 框架 | LangChain vs LangGraph vs AutoGen | LangGraph | 条件边天然支持路由，循环边天然支持Reflection |
| 向量库 | FAISS vs Qdrant vs LanceDB | LanceDB | npm 安装零部署，个人项目不需要分布式 |
| 记忆 | 手写 SQLite vs Mem0 vs Zep | Mem0 | LLM自动提取记忆，不需要预定义 Schema |
| 工具协议 | Function Calling vs MCP | MCP | 工具独立进程，跨框架复用 |
| LLM | GPT-4o vs DeepSeek vs Claude | DeepSeek | OpenAI 兼容 + 便宜 10x + 中文强 |
| 嵌入 | text-embedding-3 vs BGE-M3 | DeepSeek API | 不需要自部署模型 |
| Reranker | BGE Reranker vs Cohere | BGE Reranker V2-M3 | 开源免费 + 中英双语 + MTEB 榜首 |
| 数据库 | PostgreSQL vs MySQL vs sql.js | sql.js | 纯 JS 零依赖，单用户不需要独立服务 |

## 十、前端 Web 化

基于 Next.js 14 App Router + Vercel AI SDK 构建流式对话界面，通过 SSE 对接后端 Agent 服务。

### 10.1 流式输出

```typescript
// app/api/chat/route.ts
import { streamText } from 'ai';
import { deepseek } from '@ai-sdk/deepseek';

export async function POST(req: Request) {
  const { messages } = await req.json();
  const result = streamText({
    model: deepseek('deepseek-chat'),
    messages,
    tools: { /* MCP 工具映射 */ },
  });
  return result.toDataStreamResponse();
}
```

前端 `useChat()` 自动接 SSE——`streamText()` 返回 `DataStreamResponse`，框架层已处理 SSE 协议细节。首 Token 延迟（TTFT）< 300ms，相比同步 invoke 感知延迟降低约 80%。

### 10.2 Agent 多阶段状态管理

Agent 响应不是纯文本——包含思考、工具调用、流式回答三个阶段。前端用有限状态机管理消息生命周期：

```
SSE 事件类型 → 状态转换:
  thinking     → "正在分析问题..."
  tool_call    → "调用 arXiv 搜索..."
  tool_result  → 展示工具返回结果的预览卡片
  streaming    → 逐 Token 打字机效果
  done         → 完整回答渲染 + 引用标注
```

### 10.3 渲染性能优化

buffer 缓冲队列 + requestAnimationFrame 批量刷新：流式 chunk 高频触发 setState 导致冗余 re-render，吞吐从峰值 100+ 次/秒降至 40 次/秒以下。

Markdown 高度突变防抖：预留骨架高度 + overflow-anchor 滚动锚定 + 未闭合 Markdown 块自动补全，消除流式输出时的布局偏移。

### 10.4 安全设计

- JWT + HttpOnly Cookie 存储凭证，防止 XSS 窃取
- AI 输出通过 rehype-sanitize 白名单过滤 HTML，防止 XSS 注入
- CSP 策略：限制脚本来源和连接范围
